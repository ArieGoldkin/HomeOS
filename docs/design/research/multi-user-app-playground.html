One bot number, many families.

"Multi-tenant" reframed: not a WhatsApp number per family — one HomeOS-owned bot number, and a multi-user web app where a family is the tenant. Each member chats 1:1 with the bot (routed by phone) and logs into the UI with Google. This brief is the architecture, the build order, and the decisions to ratify.

Recommendation. Keep the single shared bot number. Adopt Better-Auth (Google OIDC), bind a member's phone with the user-sends-a-code-to-the-bot ceremony (free + most secure), and make the from_phone → family_id resolver at inbound.ts:54 the one security chokepoint. Ship D1/D2 now on single-tenancy; the D3→D4→D5 spine is the multi-user core. Own-number onboarding (D9) stays deferred per R2.

1 The reframe

What "multi-tenant" meant before R2 vs. what it means now.

✗ Before — own-number per family

  • Each family connects their own WhatsApp number (Coexistence QR).
  • Blocked: +972 Coexistence unconfirmed; group chats never reach the webhook; Business-App migration kills consumer onboarding.
  • Identity = the phone number itself, per family.

✓ After — multi-USER over one number

  • One HomeOS bot number; families route by from_phone.
  • Family = tenant; many members; each 1:1 with the bot.
  • Identity = Google login (app) + verified phone (bot), bound into one user.
Carried-over constraint (whole product). The 2026 WhatsApp Business Solution Terms restrict primary AI/LLM with an EEA + Brazil-only exception that excludes +972. HomeOS's structured "forward → event" parser most plausibly reads as ancillary (permitted) — but it's at Meta's sole discretion. Tracked in #30.

3 OTP binding ceremony

Direction B: the code travels from the member's WhatsApp → it proves control of the exact from_phone we route on. Free (user-initiated 24h window, no template).

sequenceDiagram
  autonumber
  participant M as Member (web)
  participant A as App / Server
  participant W as WhatsApp (personal)
  participant B as Bot · Cloud API
  M->>A: Google login + enter phone
  A->>A: pending_bindings(code_hash, TTL ~5m)
  A-->>M: show one-time CODE
  M->>W: type the CODE
  W->>B: send CODE (1:1 message)
  B->>A: inbound webhook (HMAC verified)
  A->>A: match from_phone + code → write family_phones
  A-->>B: bound ✓ (free service reply)
  B-->>M: confirmation in WhatsApp
  1. Member signs in with Google and enters their phone. Server writes a pending_bindings row with a high-entropy, single-use code (hashed), short TTL, attempt counter.
  2. App displays the code; member sends it to the bot from their own WhatsApp.
  3. Inbound arrives on the webhook (HMAC mandatory); the resolver matches from_phone + code against the pending claim.
  4. On match → write family_phones(family_id, from_phone, verified_at). Trust established. Bot replies "bound ✓".
  5. From now on, every 1:1 inbound from that phone resolves to that family; the board is read in the browser scoped by family_id.

4 Data model

Five net-new identity tables + three family_id migrations. credentials/oauth_state are already family-keyed — the template to copy.

users new

id, oidc_sub UNIQUE, email. Anchor on the stable OIDC subject.

families new

id. The tenant.

family_members new

user_id, family_id, role. UNIQUE(user_id, family_id). Browser-path resolver source.

family_phones new

family_id, from_phone, verified_at. UNIQUE(from_phone). Written only after OTP. Bot-path resolver source.

pending_bindings new

family_id, phone, code_hash, attempts, expires_at. Short-TTL claim, consumed once.

events migrate

No family_id today (decision A). Add column + index. Destructive deleteLastFromSender/findEventsByRef leak cross-tenant if unscoped.

conversations migrate

UNIQUE(from_phone) + INSERT OR REPLACE → cross-tenant takeover. → UNIQUE(family_id, from_phone) before family #2.

inbound_messages migrate

Add family_id; scope countFromSenderSince (G16) + statsSince. PK stays wa_message_id.

credentials · oauth_state ready

Already family_id-keyed. Only the id source changes from the FAMILY_ID constant.

7 Risk register

Ordered by blast radius. Two are hard prerequisites before family #2.

AI-ToS · +972 excluded monitor

Primary-AI restriction, EEA/Brazil-only exception. Platform-existential, Meta's sole discretion. Posture: single-purpose hard requirement; verify Anthropic ZDR. #30

Webhook HMAC was optional fixed · PR #144

Unsigned/forged POST was accepted when key unset → could bind a victim's phone to an attacker's family. Now fail-closed + boot canary.

conversations cross-tenant takeover before #2

UNIQUE(from_phone) collides across families. → UNIQUE(family_id, from_phone). #136

events destructive methods leak D3

deleteLastFromSender/findEventsByRef unscoped delete/read across tenants until family_id lands. #136

Phone reuse / transfer hijack design

Bound phone must refuse silent re-point to family B. The transfer/release authz flow is undesigned — specify before D4. #137

Group-chat ingestion constraint

Cloud API group support gated to 100k+ businesses → bot reads 1:1 only. Shapes the product, not a blocker.
